## Architecture

**This app has no traditional database.** Google Sheets is the persistence layer. Each admin's Google account gets its own spreadsheet (named `AttendanceDB`, auto-created by `initializeSpreadsheet` in [lib/sheets.ts](lib/sheets.ts)) with sheets/tabs: `courses`, `students`, `sessions`, `attendance`, `semester_config`. All reads/writes go through the Google Sheets API using the signed-in user's own OAuth token тАФ there is no service account. `lib/sheets.ts` (~1200 lines) is the entire data-access layer: every query is a full-sheet read + in-memory filter/append via `sheets.spreadsheets.values.*`.

**Auth ([auth.ts](auth.ts))**: NextAuth v5 (beta) with a single Google provider requesting `spreadsheets` + `drive.file` scopes тАФ login *is* the authorization to read/write the user's spreadsheet. The JWT callback stores `access_token`/`refresh_token`/`expires_at` and auto-refreshes the access token via `lib/google-token.ts` when it's close to expiry; a failed refresh sets `token.error = "RefreshTokenError"`, which `proxy.ts` checks on every request to force re-login. Session lifetime is 180 days (few consent prompts). On every login/refresh the refresh token is also persisted to the token registry (below) so the background scheduler can act without a browser session.

**Route protection ([proxy.ts](proxy.ts))**: everything requires an authenticated session *except* `/login`, `/check`, `/api/auth`, `/api/sheets/checkin`, `/api/line/webhook`, and `/api/cron` тАФ the student-facing check-in surface, the LINE bot webhook, and the cron-triggered scheduler tick (auth'd separately via `CRON_SECRET`, see below). Note `/projector/**` is *not* in this exempt list тАФ despite being a passive display, viewing it requires a logged-in admin session; it only shows data belonging to that admin's own spreadsheet.

**Dual persistence backends (Vercel vs local)**: production runs on **Vercel serverless**, so the three cross-request stores тАФ `lib/session-store.ts`, `lib/token-registry.ts`, `lib/line-link-store.ts` тАФ each have two backends behind one async API: **Upstash Redis** (REST client in [lib/redis.ts](lib/redis.ts), active when `UPSTASH_REDIS_REST_URL`/`UPSTASH_REDIS_REST_TOKEN` are set тАФ required on Vercel, where instances share no memory and the filesystem is read-only) and an in-process/file fallback for zero-setup local dev. Secrets persisted outside process memory (refresh tokens, session access tokens) are sealed AES-256-GCM via `lib/secret-box.ts` (key derived from `NEXTAUTH_SECRET`).

**Session lifecycle (the core domain concept)**: an admin "opens" a class session ([app/admin/session/[sessionId]](app/admin/session/%5BsessionId%5D)), which generates an OTP (`lib/otp.ts`) and GPS geofence (lat/lng/radius). Open logic is shared in `lib/session-open.ts` so the manual route and the auto-open scheduler create sessions identically. Because the check-in endpoint is public/unauthenticated, it can't hold the admin's OAuth token in a cookie тАФ instead `lib/session-store.ts` maps `sessionId тЖТ { spreadsheetId, accessToken }` (populated on open + refreshed by the admin browser's polling), plus an `OTP тЖТ sessionId` index for manual-entry mode. Entries expire after 4 hours (Redis TTL / in-memory eviction).

**Auto-open scheduler ([lib/scheduler.ts](lib/scheduler.ts))**: one guarded tick (`runSchedulerTick`) driven two ways тАФ on persistent hosts [instrumentation.ts](instrumentation.ts) starts a 60-second `setInterval` (skipped when `process.env.VERCEL` is set); on Vercel an **external cron** (e.g. cron-job.org, every minute) hits `GET /api/cron/tick`, authenticated by `CRON_SECRET` (Bearer header or `?key=`), a public path in `proxy.ts`. A Redis `SET NX EX 55` lock collapses duplicate/overlapping triggers across instances. Each tick iterates all known admins, refreshes their access token from the stored refresh token, and for courses with `SemesterConfig.auto_open_enabled` opens a session when Bangkok-local "now" falls in the teaching-schedule start window (idempotent per course+section+date+period) and auto-closes open sessions past OTP expiry (running `markAbsentStudents` on close). Schedule matching uses `lib/schedule-time.ts` (Asia/Bangkok via `Intl.DateTimeFormat` тАФ the only timezone-aware code in the repo; everything else uses server-local dates).

**Token registry ([lib/token-registry.ts](lib/token-registry.ts))**: holds each admin's Google refresh token (sealed), notification prefs (email on/off + destination + the admin's own sealed Resend API key, LINE on/off + userId), and token health status (`ok`/`invalid` тАФ invalid tokens are skipped by the scheduler and surfaced via `/api/sheets/token-status`, shown both as an admin-page banner and in the Semester Settings auto-open card). Its keys are also the scheduler's only way to discover admins (spreadsheets are per-user, so there's no global course list). Redis backend: hash `attendance:admins`; file backend: `data/admin-tokens.json` with an in-process mutex + temp-file rename. **An admin enters the registry only on a fresh login** ([auth.ts](auth.ts) persists the refresh token on sign-in and on each token refresh) тАФ sessions predating the feature never registered, which shows as `status: "unknown"`.

**Storage health check**: `GET /api/system/health` ([app/api/system/health/route.ts](app/api/system/health/route.ts)) reports whether Redis is actually configured when running on Vercel тАФ without it, `session-store`/`token-registry`/`line-link-store` silently fall back to per-instance memory/filesystem, which doesn't work across serverless instances (previously a silent failure mode with no signal to the admin). Surfaced as a non-dismissible admin-page banner (`app/admin/StorageHealthBanner.tsx`) and logged at cold start in [instrumentation.ts](instrumentation.ts).

**Notifications ([lib/notify.ts](lib/notify.ts))**: when the scheduler auto-opens a session it sends the teacher the OTP + check-in link, best-effort (failures logged both to the console and, since 2026-07-06, to `token-registry`'s `last_notify_error`/`last_notify_at`, surfaced on `/admin/notifications`). Email goes through Resend (`lib/email-notify.ts`) using a **bring-your-own-account model**: each admin registers their own free Resend account and pastes their own API key into `/admin/notifications` (guided by `app/admin/notifications/ResendGuideModal.tsx`, a dedicated modal тАФ deliberately not folded into the general `components/HelpModal.tsx`). A single shared Resend account was tried first and abandoned (2026-07-03): without a verified custom domain, Resend can only deliver to the account owner's own address, so one shared account could only ever email the developer. That restriction is a non-issue per-admin, since each admin's notification is addressed to themselves тАФ i.e. the same address their own Resend account is registered under. Sender is always Resend's shared `onboarding@resend.dev` (no per-admin "from" customization is possible without a verified domain). LINE goes through the Messaging API (`lib/line-notify.ts`, needs `LINE_CHANNEL_ACCESS_TOKEN`/`LINE_CHANNEL_SECRET`, one shared Official Account тАФ unaffected by the email model since LINE has no equivalent per-recipient restriction). LINE account linking: teacher generates a 6-char code in `/admin/notifications` (`lib/line-link-store.ts` тАФ single-use, 15-min TTL), messages it to the shared LINE Official Account, and [app/api/line/webhook/route.ts](app/api/line/webhook/route.ts) exchanges it for their LINE userId. Prefs are managed via `/api/sheets/notification-prefs` (+ `link-code`, `test-email`).

**Student check-in flow ([app/api/sheets/checkin/route.ts](app/api/sheets/checkin/route.ts))**: student hits `/check` (QR scan, carries `session_id`) or enters an OTP manually тЖТ resolves the session/access-token via `session-store.ts` тЖТ validates OTP/expiry/closed state тЖТ validates `student_id` is a 9-digit string (anti-injection) тЖТ computes GPS distance via `lib/haversine.ts` against the session's geofence тЖТ status is `present` / `late` / `gps_fail` based on distance and `late_after_min` (unless `late_enabled === false`) тЖТ duplicate check-ins for the same student+session return the existing record instead of writing a new one. Has a simple in-process IP rate limiter (10 req/min).

**Auto-flagging at check-in** (same route): records get `flagged: true`, `action_taken: "auto_flag"`, and a Thai-language `edit_note` when GPS accuracy is suspiciously poor (>100 m against a small geofence) or suspiciously perfect-and-static (spoofing heuristic), with an audit entry. Device-fingerprint conflicts are deliberately **not** auto-flagged here (removed 2026-07-06 тАФ it used to retro-edit an already-accepted *earlier* student's record with no time-gap/kiosk exemption, unfairly punishing shared-device false positives); they're instead surfaced non-destructively at review time via `lib/conflict-detection.ts`'s "confirmed" tier in the session view.

**Double periods**: a `Session` can span two periods (`period_count: 2`) with either `check_in_mode: "single"` (one check-in covers both periods) or `"double"` (two linked sessions, `linked_session_id` + `part_number` connect Part 1 тЖФ Part 2). `lib/period-utils.ts` handles period-number тЖФ time-label conversion (Thai labels, e.g. "р╕Др╕▓р╕Ъ 3тАУ4 (11:00тАУ14:30)"); periods are fixed 90-minute slots 1тАУ6 starting 08:00.

**Semester/week model**: `SemesterConfig` (per course+section) defines `semester_start`, `total_weeks`, and a `teaching_schedule` (days of week + period, plus per-day `start_time`/`end_time` and double-period settings used by auto-open). `lib/week-utils.ts` derives week numbers/labels (`W1`, `W2m`/`W2t`/... when multiple teaching days per week) from a session's date relative to `semester_start` тАФ used to build the attendance grid in course summaries. `/api/holidays` returns upcoming Thai public holidays (`date-holidays`) for schedule display.

**Anti-cheating / conflict detection ([lib/conflict-detection.ts](lib/conflict-detection.ts))**: after check-ins are collected, `buildDeviceConflicts` union-finds attendance records into clusters that share a `device_fingerprint` (UA-based) or `device_fingerprint_gpu` (canvas/WebGL, survives incognito/browser switches on the same hardware) тАФ these are "confirmed" duplicates. Records sharing only an IP + tight time/GPS proximity window are "possible" duplicates (campus WiFi can share one IP across many students). Surfaced in the admin session view for manual review/action (`approve` / `flag` / `mark_absent` / `revoke`).

**Audit trail**: attendance edits/overrides are tracked on the record itself (`edited_at/from/to`, `overridden`, `action_taken`) plus a separate `audit` log written through `app/api/sheets/audit/route.ts` for admin-driven changes (imports, manual entries, corrections, auto-flags).

**Client-side prefs**: `lib/settings.ts` keeps per-course UI settings and last-used period choices in `localStorage` (not in Sheets).

**Route structure**: `app/admin/**` (authenticated teacher UI: course list, session control, past-entry correction, roster import via `lib/xlsx-parser.ts`, semester setup with a Leaflet GPS map picker, notification settings, per-course summary grid, audit log), `app/check/**` (public student check-in), `app/projector/[sessionId]/**` (display-only live QR + roster view for classroom projection, keyed to one specific session), `app/projector/course/[courseId]/**` (stable/bookmarkable variant that polls `/api/sheets/sessions/current` for "whichever session is open right now" for a course+section, so a classroom screen left pointed at one URL keeps working across auto-opened sessions), `app/api/sheets/**` (all server routes wrapping `lib/sheets.ts`).

**Environment variables**: `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` (OAuth), `NEXTAUTH_SECRET` (also the sealing key for stored secrets, including each admin's Resend API key), `NEXTAUTH_URL` (used to build check-in links in notifications), `UPSTASH_REDIS_REST_URL` / `UPSTASH_REDIS_REST_TOKEN` (Redis backends тАФ **required on Vercel**, optional locally), `CRON_SECRET` (auth for `/api/cron/tick` тАФ without it the route returns 503 and auto-open never runs on Vercel), `NEXT_PUBLIC_GOOGLE_PICKER_API_KEY` (optional тАФ a Google Cloud **API key**, distinct from the OAuth client id/secret, restricted to the Picker API; enables "Import from Google Drive" in `app/admin/import/ImportClient.tsx`, hidden when unset), `LINE_CHANNEL_ACCESS_TOKEN` / `LINE_CHANNEL_SECRET` (optional, shared LINE Official Account for notifications тАФ missing disables the channel rather than erroring). **No server-wide Resend key** тАФ email notifications need no env var at all; each admin supplies their own Resend API key at `/admin/notifications`, sealed per-admin in the token registry (see Notifications above).

**Import from Google Drive** (`lib/drive-picker.ts`, `app/api/google/picker-token`, `app/api/sheets/import/drive-file`): lets an admin pick a roster file straight from their Drive instead of downloading it first. Google Picker is a client-side widget that needs an OAuth token in the browser to list files тАФ `/api/google/picker-token` is the one deliberate exception to "the client never sees `session.access_token`" elsewhere in this app, justified because it's still the caller's own token at the same `spreadsheets`+`drive.file` scope the server already exercises on their behalf. Once a file is picked, `/api/sheets/import/drive-file` downloads it server-side (via `alt=media`, or `files.export` to `.xlsx` for native Google Sheets) and hands the raw bytes back to the browser, which feeds them into the same `lib/xlsx-parser.ts` pipeline used for a local upload.
