## Commands

- `npm run dev` — start dev server
- `npm run build` — production build
- `npm run start` — run production build
- `npm run lint` — ESLint (flat config, `eslint-config-next` core-web-vitals + typescript)

There is no test suite/framework configured in this repo (no jest/vitest/playwright).

Stack: Next.js 16 (App Router), React 19, NextAuth v5 beta, Tailwind v4. Note Next.js 16 renamed `middleware.ts` to `proxy.ts` — this repo uses [proxy.ts](proxy.ts).

## Architecture

**This app has no traditional database.** Google Sheets is the persistence layer. Each admin's Google account gets its own spreadsheet (named `AttendanceDB`, auto-created by `initializeSpreadsheet` in [lib/sheets.ts](lib/sheets.ts)) with sheets/tabs: `courses`, `students`, `sessions`, `attendance`, `semester_config`. All reads/writes go through the Google Sheets API using the signed-in user's own OAuth token — there is no service account. `lib/sheets.ts` (~1200 lines) is the entire data-access layer: every query is a full-sheet read + in-memory filter/append via `sheets.spreadsheets.values.*`.

**Auth ([auth.ts](auth.ts))**: NextAuth v5 (beta) with a single Google provider requesting `spreadsheets` + `drive.file` scopes — login *is* the authorization to read/write the user's spreadsheet. The JWT callback stores `access_token`/`refresh_token`/`expires_at` and auto-refreshes the access token via `lib/google-token.ts` when it's close to expiry; a failed refresh sets `token.error = "RefreshTokenError"`, which `proxy.ts` checks on every request to force re-login. Session lifetime is 180 days (few consent prompts). On every login/refresh the refresh token is also persisted to the token registry (below) so the background scheduler can act without a browser session.

**Route protection ([proxy.ts](proxy.ts))**: everything requires an authenticated session *except* `/login`, `/check`, `/api/auth`, `/api/sheets/checkin`, and `/api/line/webhook` — the student-facing check-in surface plus the LINE bot webhook.

**Dual persistence backends (Vercel vs local)**: production runs on **Vercel serverless**, so the three cross-request stores — `lib/session-store.ts`, `lib/token-registry.ts`, `lib/line-link-store.ts` — each have two backends behind one async API: **Upstash Redis** (REST client in [lib/redis.ts](lib/redis.ts), active when `UPSTASH_REDIS_REST_URL`/`UPSTASH_REDIS_REST_TOKEN` are set — required on Vercel, where instances share no memory and the filesystem is read-only) and an in-process/file fallback for zero-setup local dev. Secrets persisted outside process memory (refresh tokens, session access tokens) are sealed AES-256-GCM via `lib/secret-box.ts` (key derived from `NEXTAUTH_SECRET`).

**Session lifecycle (the core domain concept)**: an admin "opens" a class session ([app/admin/session/[sessionId]](app/admin/session/%5BsessionId%5D)), which generates an OTP (`lib/otp.ts`) and GPS geofence (lat/lng/radius). Open logic is shared in `lib/session-open.ts` so the manual route and the auto-open scheduler create sessions identically. Because the check-in endpoint is public/unauthenticated, it can't hold the admin's OAuth token in a cookie — instead `lib/session-store.ts` maps `sessionId → { spreadsheetId, accessToken }` (populated on open + refreshed by the admin browser's polling), plus an `OTP → sessionId` index for manual-entry mode. Entries expire after 4 hours (Redis TTL / in-memory eviction).

**Auto-open scheduler ([lib/scheduler.ts](lib/scheduler.ts))**: one guarded tick (`runSchedulerTick`) driven two ways — on persistent hosts [instrumentation.ts](instrumentation.ts) starts a 60-second `setInterval` (skipped when `process.env.VERCEL` is set); on Vercel an **external cron** (e.g. cron-job.org, every minute) hits `GET /api/cron/tick`, authenticated by `CRON_SECRET` (Bearer header or `?key=`), a public path in `proxy.ts`. A Redis `SET NX EX 55` lock collapses duplicate/overlapping triggers across instances. Each tick iterates all known admins, refreshes their access token from the stored refresh token, and for courses with `SemesterConfig.auto_open_enabled` opens a session when Bangkok-local "now" falls in the teaching-schedule start window (idempotent per course+section+date+period) and auto-closes open sessions past OTP expiry (running `markAbsentStudents` on close). Schedule matching uses `lib/schedule-time.ts` (Asia/Bangkok via `Intl.DateTimeFormat` — the only timezone-aware code in the repo; everything else uses server-local dates).

**Token registry ([lib/token-registry.ts](lib/token-registry.ts))**: holds each admin's Google refresh token (sealed), notification prefs (email on/off + destination, LINE on/off + userId), and token health status (`ok`/`invalid` — invalid tokens are skipped by the scheduler and surfaced via `/api/sheets/token-status`, shown both as an admin-page banner and in the Semester Settings auto-open card). Its keys are also the scheduler's only way to discover admins (spreadsheets are per-user, so there's no global course list). Redis backend: hash `attendance:admins`; file backend: `data/admin-tokens.json` with an in-process mutex + temp-file rename. **An admin enters the registry only on a fresh login** ([auth.ts](auth.ts) persists the refresh token on sign-in and on each token refresh) — sessions predating the feature never registered, which shows as `status: "unknown"`.

**Notifications ([lib/notify.ts](lib/notify.ts))**: when the scheduler auto-opens a session it sends the teacher the OTP + check-in link, best-effort (failures logged, never thrown). Email goes through Resend (`lib/email-notify.ts`, needs `RESEND_API_KEY` + `NOTIFY_FROM_EMAIL`); LINE goes through the Messaging API (`lib/line-notify.ts`, needs `LINE_CHANNEL_ACCESS_TOKEN`/`LINE_CHANNEL_SECRET`). LINE account linking: teacher generates a 6-char code in `/admin/notifications` (`lib/line-link-store.ts` — single-use, 15-min TTL), messages it to the shared LINE Official Account, and [app/api/line/webhook/route.ts](app/api/line/webhook/route.ts) exchanges it for their LINE userId. Prefs are managed via `/api/sheets/notification-prefs` (+ `link-code`, `test-email`).

**Student check-in flow ([app/api/sheets/checkin/route.ts](app/api/sheets/checkin/route.ts))**: student hits `/check` (QR scan, carries `session_id`) or enters an OTP manually → resolves the session/access-token via `session-store.ts` → validates OTP/expiry/closed state → validates `student_id` is a 9-digit string (anti-injection) → computes GPS distance via `lib/haversine.ts` against the session's geofence → status is `present` / `late` / `gps_fail` based on distance and `late_after_min` (unless `late_enabled === false`) → duplicate check-ins for the same student+session return the existing record instead of writing a new one. Has a simple in-process IP rate limiter (10 req/min).

**Auto-flagging at check-in** (same route): records get `flagged: true`, `action_taken: "auto_flag"`, and a Thai-language `edit_note` when GPS accuracy is suspiciously poor (>100 m against a small geofence), suspiciously perfect-and-static (spoofing heuristic), or the device fingerprint matches another student already checked into the same session — in that case the *earlier* record is retro-flagged too, with an audit entry.

**Double periods**: a `Session` can span two periods (`period_count: 2`) with either `check_in_mode: "single"` (one check-in covers both periods) or `"double"` (two linked sessions, `linked_session_id` + `part_number` connect Part 1 ↔ Part 2). `lib/period-utils.ts` handles period-number ↔ time-label conversion (Thai labels, e.g. "คาบ 3–4 (11:00–14:30)"); periods are fixed 90-minute slots 1–6 starting 08:00.

**Semester/week model**: `SemesterConfig` (per course+section) defines `semester_start`, `total_weeks`, and a `teaching_schedule` (days of week + period, plus per-day `start_time`/`end_time` and double-period settings used by auto-open). `lib/week-utils.ts` derives week numbers/labels (`W1`, `W2m`/`W2t`/... when multiple teaching days per week) from a session's date relative to `semester_start` — used to build the attendance grid in course summaries. `/api/holidays` returns upcoming Thai public holidays (`date-holidays`) for schedule display.

**Anti-cheating / conflict detection ([lib/conflict-detection.ts](lib/conflict-detection.ts))**: after check-ins are collected, `buildDeviceConflicts` union-finds attendance records into clusters that share a `device_fingerprint` (UA-based) or `device_fingerprint_gpu` (canvas/WebGL, survives incognito/browser switches on the same hardware) — these are "confirmed" duplicates. Records sharing only an IP + tight time/GPS proximity window are "possible" duplicates (campus WiFi can share one IP across many students). Surfaced in the admin session view for manual review/action (`approve` / `flag` / `mark_absent` / `revoke`).

**Audit trail**: attendance edits/overrides are tracked on the record itself (`edited_at/from/to`, `overridden`, `action_taken`) plus a separate `audit` log written through `app/api/sheets/audit/route.ts` for admin-driven changes (imports, manual entries, corrections, auto-flags).

**Client-side prefs**: `lib/settings.ts` keeps per-course UI settings and last-used period choices in `localStorage` (not in Sheets).

**Route structure**: `app/admin/**` (authenticated teacher UI: course list, session control, past-entry correction, roster import via `lib/xlsx-parser.ts`, semester setup with a Leaflet GPS map picker, notification settings, per-course summary grid, audit log), `app/check/**` (public student check-in), `app/projector/[sessionId]/**` (public/display-only live QR + roster view for classroom projection), `app/api/sheets/**` (all server routes wrapping `lib/sheets.ts`).

**Environment variables**: `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` (OAuth), `NEXTAUTH_SECRET` (also the sealing key for stored secrets), `NEXTAUTH_URL` (used to build check-in links in notifications), `UPSTASH_REDIS_REST_URL` / `UPSTASH_REDIS_REST_TOKEN` (Redis backends — **required on Vercel**, optional locally), `CRON_SECRET` (auth for `/api/cron/tick` — without it the route returns 503 and auto-open never runs on Vercel), plus the optional notification keys above — a missing notification key disables that channel rather than erroring.
